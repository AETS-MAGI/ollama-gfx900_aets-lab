# gfx900 / MI25 ガイド

[English](./gfx900-mi25.md) | [日本語](./gfx900-mi25.ja.md)

このドキュメントは、本フォーク版 Ollama における AMD Vega / gfx900 /
Radeon Instinct MI25 固有のビルド・実行挙動を記載します。

> **スコープ:** このフォークは Linux + ROCm 環境での gfx900 推論を対象とします。
> MI25/gfx900 に対する公式な互換性保証ではありません。
> 本ドキュメントの記述は、ソースコードおよびワークログから直接確認できる事実に限定します。

---

## 1. このフォークの変更内容

upstream Ollama は gfx900/gfx906 の実行をブロックしています（コミット `55ca82726`）。
本フォークはそのブロックを解除し、gfx900 向けの安定化パッチを追加しています：

| 変更内容 | ファイル | 確認状態 |
|--------|------|--------|
| `IsROCmGFX900()` 検出メソッド追加 | `ml/device.go` | コード確認済み |
| gfx900 での Flash Attention 無効化 | `ml/device.go:FlashAttentionSupported()` | コード確認済み |
| gfx900 での `num_parallel=1` 強制 | `server/sched.go` | コード確認済み |
| gfx900 でのデフォルト `num_ctx` を 8192 にキャップ | `server/routes.go` | コード確認済み |
| `build-gfx900/lib/ollama` をパス探索で優先 | `ml/path.go` | コード確認済み |
| `CMakePresets.json` の ROCm 6 / ROCm 7 に `gfx900` 追加 | `CMakePresets.json` | コード確認済み |
| Dockerfile の rocBLAS クリーンアップを `*gfx906*` に限定（gfx900 資産を保持） | `Dockerfile` | コード確認済み |

## 2. 前提条件

- Linux（Ubuntu 22.04 または 24.04 推奨）
- ROCm 6.1 以降インストール済み（`amdgpu-install`）
- `rocminfo` で `gfx900` デバイスが認識されていること
- Go 1.24.1 以上
- CMake 3.21 以上、HIP コンパイラ（ROCm に付属）

デバイス認識の確認：

```bash
rocminfo | grep -E "Name|gfx"
```

以下のような出力が得られれば認識されています：

```
  Name:                    gfx900
```

## 3. ビルド

### 3.1 gfx900 向け Ollama のビルド

gfx900 固有パッチがすべて含まれる `vega-int8-probe` ブランチを使用します：

```bash
git checkout vega-int8-probe
```

gfx900 ターゲットを明示してビルドします：

```bash
cmake -B build-gfx900 \
    -DAMDGPU_TARGETS=gfx900 \
    -DCMAKE_BUILD_TYPE=Release
cmake --build build-gfx900 --parallel $(nproc)
```

ビルド成果物は `build-gfx900/lib/ollama/` に配置されます。
`ml/path.go` のパス探索ロジックがこのディレクトリを優先します。

### 3.2 gfx900 向け rocBLAS のビルド（必要な場合）

システムの ROCm パッケージには gfx900 向け Tensile カーネルが含まれていない場合があります。
rocBLAS の初期化に失敗する場合や性能が不十分な場合は、関連リポジトリ
[ROCm-MI25-build](https://github.com/AETS-MAGI/ROCm-MI25-build) のスクリプトを使用してローカルビルドを行います：

```bash
# ROCm-MI25-build リポジトリのルートから実行
./build-rocblas-gfx900.sh
```

実行時にライブラリパスを注入します：

```bash
export ROCBLAS_TENSILE_LIBPATH=/path/to/local/rocblas/library
```

## 4. 実行時の挙動

### 4.1 gfx900 の自動検出

サーバー起動時に `compute=gfx900` の ROCm デバイスが検出されると、
以下が自動的に適用されます（環境変数の手動設定は不要）：

- **Flash Attention**: `FlashAttentionSupported()` が gfx900 で `false` を返すため（`ml/device.go`）、`OLLAMA_FLASH_ATTENTION` の設定にかかわらず Flash Attention は使用されません。ログには `FlashAttention:Auto`（パラメータ型の初期値）と表示されますが、これは正常な動作です。
- **`num_parallel`**: 1 に強制（`server/sched.go`）
- **`num_ctx` デフォルト値**: VRAM 基準のデフォルトが 8192 を超える場合のみ 8192 にキャップ（`server/routes.go`）。MI25（16 GB HBM2）では VRAM 基準のデフォルトは通常 4096 であり、キャップは適用されません。

gfx900 検出時のログ出力例：

```
level=INFO msg="forcing num_parallel=1 for ROCm gfx900" gpu=... compute=gfx900
level=INFO msg="capping default context for ROCm gfx900" default_num_ctx=8192
```

### 4.2 デバイスの可視化制御

MI25 のみに Ollama を使用させるには：

```bash
export ROCR_VISIBLE_DEVICES=<UUID>
```

UUID の確認方法：

```bash
rocminfo | grep -A5 "gfx900" | grep "Uuid"
```

### 4.3 ライブラリパス探索順序

開発時、ランタイムは以下の順序でバックエンドを探索します（`ml/path.go`）：

1. `<実行ファイルディレクトリ>/build-gfx900/lib/ollama`
2. `<カレントディレクトリ>/build-gfx900/lib/ollama`
3. 配布パス（`../lib/ollama` など）
4. `<実行ファイルディレクトリ>/build/lib/ollama`
5. `<カレントディレクトリ>/build/lib/ollama`

明示的に上書きするには：

```bash
export OLLAMA_LIBRARY_PATH=/your/custom/path
```

## 5. 既知の制限

| 制限事項 | 理由 | 根拠（コード） |
|----------|------|--------------|
| Flash Attention が使用されない | `FlashAttentionSupported()` が gfx900 で `false` を返す。`OLLAMA_FLASH_ATTENTION` は無効 | `ml/device.go` |
| 同時リクエスト数: 最大 1 | `num_parallel` が強制的に 1 | `server/sched.go` |
| `num_ctx` 上限: 8192 | VRAM 基準のデフォルトが 8192 超の場合のみ適用。MI25（16 GB）では通常デフォルト 4096 でキャップ不適用 | `server/routes.go` |
| Windows 非対応 | upstream の Windows HIP が gfx900 を除外 | `CMakeLists.txt` |
| MLIR iGEMM 利用不可 | upstream で gfx900 向けが無効化（コミット `2407d2f`、2021年） | MIOpen ソース |
| rocBLAS Tensile カーネルが存在しない場合あり | 標準 ROCm パッケージに gfx900 向けが含まれないことがある | 実行時観察 |
